# 分析指定的字段的计算逻辑

## 适用场景
- 用户提供一个代码入口（Controller/Service 方法等）要求分析其输出字段以及内部变更字段的具体**计算逻辑**
- 用户提供其它基础分析文档，要求你继续挖掘字段的**计算逻辑**
- 用户要求：
  - 字段级来源拆解（直接输入/查询/计算/覆盖）；
  - 复杂字段补充；
  - 按复杂度标记重点关注项；
  - 保持文档整体结构与可读性。


## 分析思路
1. 先分析指定的代码链路中所有的影响字段（有可能给出了基线文档，那就参考基线文档），得到所有字段的集合
2. 结合所有输出字段，以及代码链路，分析字段的计算逻辑。看看所有输出受哪些输入、内部的逻辑影响。比如一个字段的生成是输入的原始字段直接得到的，还是在另一个表中查询得到的，又或者经过一些特定链路计算后得到的。有的字段比较简单，有的字段比较复杂，复杂的字段可能经过很多层，都需要梳理出来。
   - 代码分析需要覆盖所有你可以触及的代码（第三方包默认不分析，除非用户要求）
3. 分析完成后，需要再次检查你的分析，是否将计算逻辑展开了，如果未展开请再次阅读代码，补充细节
4. 当执行完上述步骤后，按照输出模版输出字段的计算逻辑
5. 最后再检查当前文档是否还有没有解释清楚的字段，按照‘复杂度评级标准’标记它们（具体标准见后续章节）


## 分析口径（字段级）

对每个字段至少给出以下 6 项中的 4 项（复杂字段尽量给全）：

1. 字段落点（返回体/表名/实体字段）
2. 输入来源（请求、登录态、配置、数据库、外部系统）
3. 生成逻辑（拼接、映射、聚合、排序、阈值、时间线）
4. 覆盖顺序（首次赋值、二次修正、最终主导来源）
5. 默认值/兜底/异常分支
6. 风险与验证建议

## 复杂度评级标准

- `L1-低`：直接透传或简单常量赋值。
- `L2-中`：单层计算或单次映射，无覆盖。
- `L3-高`：多输入融合、条件分支明显，或存在一次以上覆盖。
- `L4-极高`：跨方法/跨阶段策略链路，存在时间线、排序选取、口径不对称、反向回写等。

## 输出模板（固定）

```markdown
## 0) 概要总结
- 这段代码在做什么（2~5 句）：
- 业务价值与边界（做什么/不做什么）：
- 与本次字段分析的关系（为什么要分析这些字段）：

## 1) 入口与基线
- 入口：
- 基线文档：
- 本次补充目标：

## 2) 输出汇总
- 按照分类（比如表、作用域等）、操作类型、备注，进行汇总描述

## 3) 输出链路详解（字段级）
- 按照分类对各个字段按照‘分析口径（字段级）’尽可能的分析字段


## 4) 复杂字段审计与关注优先级
- 检查‘输出链路详解（字段级）’的字段，按照‘复杂度评级标准’对字段进行评级，列出所有L3、L4的字段，并说明其“当前展开充分性”、“为什么复杂”、“还缺的细节”
- 最后给出需要重点关注的字段

```

## 质量检查清单

- [ ] 开头包含“概要总结”，并说明本次字段分析范围与目标
- [ ] 概要总结聚焦职责与边界，未与“输出链路详解”重复实现细节
- [ ] 已读取“基线文档 + 入口代码”
- [ ] 分析逻辑覆盖所有可触达的代码
- [ ] 每个 L3、L4 字段都给出具体复杂原因

## 禁止事项

- 禁止脱离基线文档直接重写全篇。
- 禁止仅给结论不列字段。
- 禁止把普通日志当业务输出字段。
- 禁止把“可能”当“已证实”，未闭环必须标记 `Unknown`。


# 落盘规则

- 默认将结果写成 Markdown 落盘至仓库 `ai_docs/`。
- 文件名遵循 `ai-docs-md-naming`（`YYMMDDNN-描述.md`）。
- 若用户要求仅对话输出、禁止生成文件或已指定路径，则以用户要求为准。
