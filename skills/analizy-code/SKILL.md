# 代码入口解读（通用版）

## 何时启用与执行约束

触发词与适用场景以 YAML **`description`** 为准；技能加载后须遵守：

- **无锚点**：先追问入口（哪个文件、哪一段或哪个符号），再解读。
- **有锚点**：按下文「输出结构」完整展开，**禁止**仅给笼统概括。
- **常见入口类型**：HTTP/接口（含 Java API）、应用服务方法、消息消费者、定时任务、CLI/脚本入口等。

---

## 落盘文档时的格式与文件命名

**默认落盘**（须同时满足）

- 已完成一次**完整**解读（即使用户未说「保存」「落盘」）。
- 用户未声明「不要生成文件」或「只在对话里简要说明」。

此时应将解读结果写入仓库内文档。若用户**未**指定文件名或路径：

1. 文件类型 **必须为** Markdown（扩展名 `.md`）。
2. 目录与文件名 **必须** 按 [ai-docs-md-naming/SKILL.md](../ai-docs-md-naming/SKILL.md) 执行：默认目录为仓库根下 **`ai_docs/`**，文件名为 **`YYMMDDNN-文档描述.md`**（列出目录 → 按当日 `YYMMDD` 取已有最大 `NN` → `+1`）。

**例外（以用户为准）**

- **不要生成文件**，或只要**对话中简要说明**：仅在对话中输出，不落盘。
- **用户指定了输出路径或文件名**（是否在 `ai_docs/`、是否 `.md` 均可）：按用户指定执行，不再强制套用 `ai-docs-md-naming` 的默认目录与序号；若指定非 `.md`，则不强制 Markdown。

## 输出结构（必须覆盖以下模块）

按顺序组织；某模块不适用时写 **「本入口不涉及」** 并一句话说明原因。文档目标是：**准确分析指定代码的输入、输出及中间转换的核心逻辑**，并按下列结构展现。

### 1. 入口与入参解析

侧重「如何能进到这条链路」与「入参如何驱动流程」：

| 内容 | 说明 |
|------|------|
| **入口说明** | 入口类型（HTTP、应用服务方法、MQ 消费者、定时任务、CLI/main 等）及在系统中的角色（一句）。 |
| **进入条件** | 触发时机、路由/订阅匹配、前置校验、网关或过滤器要求；**影响能否执行** 的条件单独点出。 |
| **入参形态与格式** | 请求体、Query、Path、消息体、命令行参数等；关键类型或 DTO/结构名。 |
| **参数逻辑** | **必要参数**；**影响分支、校验、幂等、权限** 的字段；默认值、枚举、互斥或组合条件；不必罗列无关字段。 |

**若为 Java / Spring REST 接口**，在上一小节基础上补充 **接口锚定**（列表或短表均可）：

- **HTTP**：方法与 `@GetMapping` / `@PostMapping` 等；**完整路径** = 类级 `@RequestMapping` + 方法级 path。
- **文档名**：`@ApiOperation` 或 OpenAPI 注解中的摘要。
- **权限与安全**：`@PreAuthorize`、`@ValidLoginUser`、过滤器链等；简述生效条件（可引用 SpEL 含义）。
- **入参绑定**：`@RequestBody` / `@RequestParam` / `@PathVariable` 及类型；**返回值** 包装类型（如 `Result<T>`）。

---

### 2. 出参、数据变更与外部依赖

侧重「对外返回什么」与「对内对外产生了哪些状态变化」：

| 内容 | 说明 |
|------|------|
| **出参** | 返回值、响应体、错误码结构；只强调 **影响调用方理解业务结果** 的字段。 |
| **存储变更** | 数据库表 / 文档集合 / 缓存键等：**读 / 写 / 删**；若代码仅能推断，标 **待确认**。 |
| **外部依赖** | 其它微服务、第三方 HTTP、RPC、MQ 生产与消费等。对每个依赖尽量判定 **读型** 还是 **可能写状态**（依据方法语义、HTTP 动词、事务边界、幂等等）；不确定则写 **待确认**。 |
| **其它副作用** | 发消息、审计日志、文件 IO 等（一句话即可）。 |

---

### 3. 流程（重点模块）

流程章节的结构与质量决定整篇分析是否可用；须做到 **概括 → 图示与文字细化 → 不遗漏小分支**。

#### 3.1 核心流程概括

用 **3～8 句话** 概括「先做什么、再做什么、最终达成什么」：**不写实现细节**（细节放在 3.2 与图中）；表述便于非实现者快速理解。

#### 3.2 流程细化（纵向链路 + 文字）

从 **入口方法体** 出发，用搜索与阅读 **逐级下钻**，直到 **持久化、外部 IO、或明确边界**：

- **通用**：入口 → 直接调用的模块/类/函数 → 下层；每一步写 **`模块或文件名.符号名`** + **一句职责**。
- **典型分层（若项目存在）**：适配层（Controller/Handler）→ 应用/领域服务 → 仓储/DAO/客户端 → DB/缓存/消息/第三方 API。
- **原则**：优先 **成功主路径**；次要分支可简要单列，避免堆砌无关调用。若对单一路径的文字说明 **超过约 150 字**、或单张流程图 **节点过多**，应拆为 **父子流程**（先总览再子图）。除主流程及相关子流程外，**易忽略的短小分支**（如补偿、降级回调）也应用 **独立小流程图** 呈现，防止遗漏细节。

**若为 Java 后端接口**，建议按层命名（与项目一致即可）：

1. **Controller** → 注入的 `*Service` / `*Manager`。
2. **`*ServiceImpl` 等** → Manager、工具类、远程客户端。
3. **持久化**：MyBatis Mapper + XML，或 JPA Repository → **SQL / 表名**。
4. **外部系统**：HTTP/RPC/MQ —— 在步骤中 **单独标出**。

#### 3.3 流程图 / 时序图（规则）

| 规则 | 说明 |
|------|------|
| **何时画** | 当仅用文字描述主路径会 **超过约 100 字**、或存在 **并行、重试、异步、多参与者** 时，必须配图。 |
| **形式** | **流程图**（判断与分支）、**时序图**（多组件交互）、或二者结合。使用 **Mermaid**（`flowchart`、`sequenceDiagram`）等可渲染语法。 |
| **粒度** | **禁止** 用单张巨型图塞满所有细节。采用 **父子流程**：先一张 **总览图**（5～10 个节点），再为关键子过程各配 **子图**（命名如「子流程 A：鉴权与限流」）。 |
| **与正文关系** | 每张图前用 **2～4 句话** 说明该图覆盖的范围；图中节点名与 3.2 中的 **类.方法** 或模块名 **对齐**。 |
| **Mermaid 兼容** | `flowchart` 中带标签的分支边：箭头与标签须写成 **`-->` 与两侧 `|标签|` 的组合**（与 Mermaid 文档一致），**不要**写成「`--` + 文字 + `-->`」这种旧式写法（多数渲染器无法解析）。节点文案含冒号、括号、斜杠时，优先用 **`id["短文本"]`**。`erDiagram` 的关系说明中避免 **`&`**，改用「与」或拆句。 |

### 4. 实体清单与关系（重点模块）

当本入口涉及 **读写存储、缓存、消息载荷、或跨层 DTO 转换** 时，必须包含：

| 小节 | 内容 |
|------|------|
| **关键实体清单** | 表名 / 文档集合 / 消息 Topic / 核心领域对象；每个用一行说明 **在本流程中的用途**（读/写/转换）。 |
| **关系与约束** | 实体间 **一对多 / 多对一 / 多对多**；**中间表**；**外键或关联字段**（能从代码、注解、注释推断则写；否则标 **待确认**）。 |
| **关系图** | 至少一种：**ER 图**（Mermaid `erDiagram` 或等价）或 **实体依赖简图**；复杂时拆成 **主 ER + 局部展开**。每个实体在图中**不超过约 3 个代表性字段**（如主键、业务主键、状态字段等），其余字段放在「字段说明」小节中单独列出。 |
| **数据转换 / 数据流** | 请求 DTO → 领域对象 → 持久化结构 → 响应 DTO 的 **转换链**；可用 **小型示意图**（方框+箭头）或表格「源字段 → 规则/映射 → 目标字段」。 |
| **字段说明** | 对 **影响业务规则、校验、落库、幂等键** 的字段做表格式或列表说明（**不必** 罗列所有列）；注明 **枚举含义、默认值、与外部系统字段对应关系**（若有）。 |

若仅有 SQL 字符串、无实体类，仍列出 **逻辑表/视图** 及用途。实体间 **转换与依赖** 须用 **实体关系图 + 文字** 描述清楚。

---

### 5. 用例

至少 **一个** 完整用例，帮助读者理解「这条入口在实际执行时发生了什么」：

- **入参**：给出具体（可示例化）的字段值或消息体片段；敏感信息可脱敏并标注「示例」。
- **内部关键变化**：结合代码路径说明 **主分支如何走**、**查/写了哪些数据或调了哪些外部接口**（与第 2 节呼应）。
- **出参**：示例化返回或状态变化结果。纯查询则说明结果如何随入参变化；写入则说明入参与最终落库/对外状态的关系。

### 6. 其它技术与横切关注点（收尾；可主动检查）

放在全文 **最后** 分析：除 **不确定性与缺口** 外，由 AI **主动检索** 并在本节简述（**有则写，无则省略**），避免喧宾夺主。

| 类别 | 示例 |
|------|------|
| **横切** | AOP、拦截器、过滤器、中间件、全局异常处理、统一日志/审计、幂等装饰器。 |
| **并发与一致性** | 事务边界（本地/分布式）、锁、乐观版本、异步与回调顺序。 |
| **安全** | 认证鉴权、数据权限、脱敏、密钥与配置来源。 |
| **可靠性与运维** | 重试、超时、熔断、降级、死信、监控埋点。 |
| **不确定 / 待确认** | 若无法从代码确定表关系、动态 SQL、配置驱动分支：写 **「待确认」** + **缺少什么信息**（如无 Mapper、仅有字符串拼接 SQL）。 |

## 语言 / 栈专项速查

### Java / Spring REST（接口解读时叠加）

- 追踪时优先 **Grep 方法名** 定位 `*ServiceImpl`，再下钻 Mapper。
- Mapper：**接口 + XML** 中的 `id`、表名、`resultMap`。

### 其它栈

- **Python**：入口装饰器/路由、依赖注入、ORM Model ↔ 表。
- **前端**：路由、状态管理、API 层、与后端契约字段。
- **脚本/CLI**：参数解析、配置文件、外部命令调用链。

按项目实际 **替换层名称**，不强行套用 Java 包名。

## 自检清单

- [ ] **第 1 节** 写清进入条件、入参形态及 **影响流程的参数逻辑**；Java 接口已做路径与绑定锚定。
- [ ] **第 2 节** 写清出参，并对 **存储变更与外部依赖** 区分读/写倾向或标 **待确认**。
- [ ] **第 3.1** 为 3～8 句概括，**不与 3.2 细节重复**；**3.2** 至少到达 **业务服务层 + 数据访问或明确无落库**；过长文字或过大单图已拆 **父子流程**，小分支有独立图。
- [ ] **3.3** 满足配图规则时已用 Mermaid；每张图前有范围说明，节点与 **3.2** 中的符号对齐。
- [ ] **第 4 节** 有数据参与时含 **实体清单 + 关系或转换说明**；**ER 图中每实体约 ≤3 个字段**，其余在「字段说明」中说明。
- [ ] **第 5 节** 至少 **一条** 完整用例（入参 → 内部变化 → 出参）。
- [ ] **第 6 节** 置于文末；对行为有明显影响的横切已主动检查；缺口已标 **待确认**。
- [ ] 若用户**未**显式说明「不要生成文件」或「改为写入其它路径」，解读结果已落盘为 **`.md`** 文档：
  - 未指定文件名或路径时：写入默认 **`ai_docs/`**，并按 **ai-docs-md-naming** 生成 `YYMMDDNN-描述.md`；
  - 已指定文件名或路径时：严格遵循用户指定。
